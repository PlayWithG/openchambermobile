# Technical Design: OpenChamber Android App

## Overview

This design covers the technical architecture for delivering OpenChamber as a native Android application using Capacitor, with a dedicated connectivity management layer that makes mobile sessions robust under foreground/background transitions, network type changes, and remote server loss.

**Core Design Principle:** The mobile connectivity layer is the primary deliverable — NOT the UI wrapper. The UI already works on mobile; the hard problem is making SSE connections resilient on a device that changes networks, sleeps, and resumes unpredictably.

---

## 1. Architecture Decisions

### 1.1 Capacitor Wrapper Structure

**Decision: `packages/mobile` as a thin Capacitor shell, NOT a fork of `packages/ui`.**

```
packages/mobile/                 # New workspace — thin Capacitor shell
├── capacitor.config.ts          # Web dir points to packages/ui build output
├── package.json                 # Workspace dep on @opencode-ai/ui
├── android/                     # Generated by `npx cap add android`
│   ├── app/
│   │   ├── src/main/
│   │   │   ├── java/.../
│   │   │   │   └── MobileApp.kt   # Minimal MainActivity — no custom native code
│   │   │   ├── res/

│   │   │   └── AndroidManifest.xml
│   │   └── build.gradle
│   └── build.gradle
├── src/
│   ├── mobile-entry.ts          # Mobile-specific bootstrap (registers plugins)

│   └── session-storage.ts        # Capacitor Preferences bridge
```

**Why NOT deeper integration into android/:** Capacitor generates and manages `android/`. Custom native code should be minimal — the spec explicitly excludes custom native plugins. All logic lives in TypeScript inside `packages/mobile/src/` and `packages/ui/src/lib/`.

**Boundary rules:**

- `packages/mobile/src/` only bridges Capacitor plugins to platform-agnostic interfaces.
- `packages/ui/src/lib/opencode/connectivity.ts` contains ALL connectivity logic — it uses injected adapter interfaces, NOT Capacitor APIs directly.
- `packages/ui/src/lib/opencode/session-persistence.ts` uses an injected storage adapter, NOT Capacitor Preferences directly.
- This separation means the connectivity and persistence modules work in desktop/web without Capacitor present.

### 1.3 Platform Detection

All platform-specific code flows through a single detection point:

```typescript
// packages/ui/src/lib/opencode/platform-detector.ts

export type Platform = 'web' | 'desktop' | 'mobile';

export interface PlatformCapabilities {
  platform: Platform;
  isMobile: boolean;
  hasNetworkPlugin: boolean;
  hasAppPlugin: boolean;
  hasPreferencesPlugin: boolean;
}

let cached: PlatformCapabilities | null = null;

export function detectPlatform(): PlatformCapabilities {
  if (cached) return cached;

  const win = window as typeof window & {
    __OPENCHAMBER_MOBILE__?: true;
    __OPENCHAMBER_DESKTOP_SERVER__?: unknown;
    __OPENCHAMBER_RUNTIME_APIS__?: { runtime?: { isDesktop?: boolean } };
  };

  const isDesktop = Boolean(win.__OPENCHAMBER_RUNTIME_APIS__?.runtime?.isDesktop);
  const isMobile = Boolean(win.__OPENCHAMBER_MOBILE__);

  cached = {
    platform: isMobile ? 'mobile' : isDesktop ? 'desktop' : 'web',
    isMobile,
    hasNetworkPlugin: isMobile,
    hasAppPlugin: isMobile,
    hasPreferencesPlugin: isMobile,
  };

  return cached;
}
```

The mobile entry point sets `window.__OPENCHAMBER_MOBILE__ = true` before the React app boots. This follows the existing pattern used by `__OPENCHAMBER_DESKTOP_SERVER__` in `client.ts`.

## 2. Data Flow: SSE Events → Connectivity Manager → UI

### 2.1 Current Architecture (Web/Desktop)

```
SDK global.event() → event-pipeline.ts → onEvent callback → sync stores → React
                     ↑
                     heartbeat timeout + visibility change
                     (reconnect via loop)
```

The existing `event-pipeline.ts` already handles:
- SSE stream iteration with abort control
- Heartbeat timeout detection (15s default)
- Visibility change → abort if heartbeat is stale → reconnect
- bfcache pageshow → abort → reconnect
- Coalescing of same-type events
- 250ms reconnect delay

### 2.2 Mobile Architecture (New)

```
Capacitor App Plugin ─── foreground/background events
       │
Capacitor Network Plugin ── connectivity change events
       │
       ▼
ConnectivityManager (packages/ui/src/lib/opencode/connectivity.ts)
       │
       ├── on foreground → validate liveness → reconnect SSE
       ├── on network change → tear down SSE → reconnect on new network
       ├── on disconnect detected → exponential backoff retry
       │
       ▼
useConnectionStore (Zustand — leaf selectors)
       │
       ├─ status: 'connected' | 'reconnecting' | 'disconnected'
       ├─ lastConnectedAt: number | null
       ├─ reconnectAttempt: number
       ├─ networkType: 'wifi' | 'cellular' | 'unknown' | 'none'
       │
       ▼
ConnectionStatusBar → reads status via leaf selector
```

### 2.3 Integration with Existing Event Pipeline

**Key Decision: ConnectivityManager wraps the event pipeline, it does NOT replace it.**

The `event-pipeline.ts` continues to handle SSE stream iteration, coalescing, and flushing. The `ConnectivityManager` provides an additional lifecycle layer that:

1. **Listens to Capacitor events** (foreground/background, network changes) and acts on the SSE connection.
2. **Validates liveness before restoring** the SSE stream after background.
3. **Manages exponential backoff** when the remote server is unreachable (the existing pipeline just does a fixed 250ms reconnect).

The integration point:

```typescript
// Modified event-pipeline.ts hook points
export type EventPipelineInput = {
  sdk: OpencodeClient
  onEvent: (directory: string, payload: Event) => void
  onReconnect?: () => void
  // NEW: Mobile connectivity hooks
  onStreamConnected?: () => void
  onStreamError?: (error: unknown) => void
  shouldReconnect?: () => boolean             // Ask connectivity manager before reconnecting
  getReconnectDelay?: () => number             // Override reconnect delay (backoff)
}
```

When `platform === 'mobile'`, `ConnectivityManager` provides `shouldReconnect` and `getReconnectDelay` callbacks. When `platform === 'web' | 'desktop'`, these hooks are absent and the pipeline behaves exactly as before (250ms fixed delay).

**This is a non-breaking change.** The event pipeline's core loop is untouched — only two new optional hooks are added.

## 3. ConnectivityManager Design

### 3.1 Class Structure

```typescript
// packages/ui/src/lib/opencode/connectivity.ts

export type ConnectionState = 'connected' | 'reconnecting' | 'disconnected';

export type NetworkType = 'wifi' | 'cellular' | 'unknown' | 'none';

export interface ConnectivityAdapter {
  onAppResume: (callback: () => void) => () => void;
  onAppPause: (callback: () => void) => () => void;
  onNetworkChange: (callback: (networkType: NetworkType) => void) => () => void;
  getCurrentNetworkType: () => NetworkType;
}

export interface StorageAdapter {
  get: (key: string) => Promise<string | null>;
  set: (key: string, value: string) => Promise<void>;
  remove: (key: string) => Promise<void>;
}

export interface LivenessCheckResult {
  alive: boolean;
  serverReachable: boolean;
}

export class ConnectivityManager {
  private state: ConnectionState;
  private adapter: ConnectivityAdapter | null;
  private networkType: NetworkType;
  private backoff: ExponentialBackoff;
  private isBackgrounded: boolean;
  private cleanupFns: Array<() => void>;
  private livenessUrl: string;

  // Public state access (for store sync)
  private onStateChange: (state: ConnectionState) => void;

  constructor(config: ConnectivityManagerConfig);

  // Lifecycle
  start(): void;
  stop(): void;

  // Called by event-pipeline hooks
  onStreamConnected(): void;
  onStreamError(error: unknown): void;
  shouldReconnect(): boolean;
  getReconnectDelay(): number;

  // Manual reconnect (from UI "Retry" button)
  reconnect(): void;

  // State queries
  getState(): ConnectionState;
  getNetworkType(): NetworkType;
}

class ExponentialBackoff {
  private attempt: number;
  private readonly baseMs: number;    // 1000 (1s)
  private readonly maxMs: number;     // 30000 (30s)
  private readonly jitter: number;    // 0.2 (±20% randomization)

  next(): number;    // Returns delay in ms with jitter
  reset(): void;     // Reset to attempt 0
  peek(): number;    // Peek at next delay without incrementing
}
```

### 3.2 Event Handling Flow

```
┌──────────────────────────────────────────────────────────────┐
│                    Foreground / Background                    │
│                                                              │
│  App.pause event ──► isBackgrounded = true                  │
│                       state = 'reconnecting'                 │
│                       (keep SSE alive — WebView policy)       │
│                                                              │
│  App.resume event ──► isBackgrounded = false                 │
│                       state = 'reconnecting'                  │
│                       ┌─────────────────────────┐            │
│                       │ Liveness Check           │            │
│                       │  GET /health              │            │
│                       │  ┌─────────┐ ┌────────┐ │            │
│                       │  │ alive   │ │ stale   │ │            │
│                       │  │ resume  │ │ discard │ │            │
│                       │  │ SSE     │ │ +fresh  │ │            │
│                       │  └─────────┘ └────────┘ │            │
│                       └─────────────────────────┘            │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                    Network Transition                          │
│                                                              │
│  Network.change ──► newNetworkType captured                  │
│                     IF networkType === 'none'                 │
│                       state = 'disconnected'                  │
│                       (no retry — no network)                 │
│                     ELSE                                      │
│                       state = 'reconnecting'                  │
│                       backoff.reset()                         │
│                       attempt reconnect on new network        │
│                       (5s max to establish, per spec)         │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                    Remote Server Loss                          │
│                                                              │
│  SSE error / heartbeat timeout ──► state = 'reconnecting'    │
│                                    backoff.next() → delay     │
│                                    retry after delay           │
│                                                              │
│  Liveness check fails ──► state = 'disconnected'              │
│                           (retry requires manual tap          │
│                            or connectivity restore)           │
└──────────────────────────────────────────────────────────────┘
```

### 3.3 Exponential Backoff Algorithm

```
delay(attempt) = min(baseMs * 2^attempt, maxMs) ± jitter

Where:
  baseMs  = 1000  (1 second)
  maxMs   = 30000 (30 seconds)
  jitter  = ±20% randomization (prevents thundering herd)

Sequence (without jitter):
  attempt 0:  1s
  attempt 1:  2s
  attempt 2:  4s
  attempt 3:  8s
  attempt 4:  16s
  attempt 5+: 30s (capped)

On successful connection → reset back to attempt 0.
On network change → reset back to attempt 0 (new network, fresh start).
On manual reconnect → reset back to attempt 0.
On foreground → start at attempt 0 (liveness check first).
```

### 3.4 Liveness Validation

When the app resumes from background, BEFORE restoring the SSE stream:

1. **Attempt liveness check:** `GET /health` to the remote OpenCode server.
2. **Timeout:** 5 seconds. If the server doesn't respond within 5 seconds, consider it unreachable.
3. **If alive:** Resume SSE with the existing session. The event pipeline reconnects, and `onReconnect` fires to refresh state.
4. **If stale or unreachable:** Discard stale session state. Set state to `reconnecting`. Begin exponential backoff to re-establish connection. The sync store's `bootstrap` runs fresh once the connection is re-established.

This prevents "zombie sessions" where the UI shows stale data that the server has already discarded.

### 3.5 Capacitor Plugin Integration

```typescript
// packages/mobile/src/connectivity-plugin.ts

import { App } from '@capacitor/app';
import { Network } from '@capacitor/network';
import type { ConnectivityAdapter, NetworkType } from '@opencode-ai/ui';

export function createCapacitorConnectivityAdapter(): ConnectivityAdapter {
  return {
    onAppResume(callback) {
      App.addListener('appStateChange', (state) => {
        if (state.isActive) callback();
      });
      return () => App.removeAllListeners('appStateChange');
    },

    onAppPause(callback) {
      App.addListener('appStateChange', (state) => {
        if (!state.isActive) callback();
      });
      return () => App.removeAllListeners('appStateChange');
    },

    onNetworkChange(callback) {
      Network.addListener('networkStatusChange', (status) => {
        const type = mapCapacitorNetworkType(status.connectionType);
        callback(type);
      });
      return () => Network.removeAllListeners('networkStatusChange');
    },

    getCurrentNetworkType() {
      // Synchronous — returns last known state
      // Capacitor's Network.getStatus() is async, so we cache
      return cachedNetworkType;
    },
  };
}

type CapacitorConnectionType = 'wifi' | 'cellular' | 'unknown' | 'none' | 'bluetooth' | 'ethernet' | 'mixed';

function mapCapacitorNetworkType(type: CapacitorConnectionType): NetworkType {
  switch (type) {
    case 'wifi': return 'wifi';
    case 'cellular': return 'cellular';
    case 'none': return 'none';
    default: return 'unknown';
  }
}
```

**Important:** `ConnectivityAdapter` is an interface injected at construction time. The `ConnectivityManager` in `packages/ui` has ZERO Capacitor imports. This means:
- Desktop/web builds don't need `@capacitor/*` packages.
- The manager is fully testable with a mock adapter.
- Different platforms can supply different adapters.

## 4. Session Persistence Design

### 4.1 Storage Architecture

```typescript
// packages/ui/src/lib/opencode/session-persistence.ts

export interface SessionData {
  endpointUrl: string;           // e.g., 'http://192.168.1.100:4096'
  authToken: string | null;       // Bearer token or API key
  activeSessionId: string | null; // Last active session ID
  lastDirectory: string | null;   // Last working directory
  lastConnectedAt: number;        // Timestamp of last successful connection
}

export interface SessionPersistence {
  save(data: SessionData): Promise<void>;
  load(): Promise<SessionData | null>;
  clear(): Promise<void>;
}
```

### 4.2 Capacitor Preferences Implementation

```typescript
// packages/mobile/src/session-storage.ts

import { Preferences } from '@capacitor/preferences';
import type { StorageAdapter } from '@opencode-ai/ui';

const STORAGE_KEYS = {
  endpointUrl: 'oc_endpoint_url',
  authToken: 'oc_auth_token',
  activeSessionId: 'oc_active_session_id',
  lastDirectory: 'oc_last_directory',
  lastConnectedAt: 'oc_last_connected_at',
} as const;

export function createCapacitorStorageAdapter(): StorageAdapter {
  return {
    async get(key: string): Promise<string | null> {
      const { value } = await Preferences.get({ key });
      return value;
    },
    async set(key: string, value: string): Promise<void> {
      await Preferences.set({ key, value });
    },
    async remove(key: string): Promise<void> {
      await Preferences.remove({ key });
    },
  };
}
```

**Why Capacitor Preferences and NOT localStorage:**
- localStorage is ephemeral on Android WebView — the OS can clear it under memory pressure.
- Capacitor Preferences uses SharedPreferences on Android (persistent across app restarts).
- Desktop/web fallback uses localStorage (no Capacitor = no Preferences API).

### 4.3 Data Schema

Each value is stored as a separate key (NOT a single JSON blob) for two reasons:
1. **Atomic updates:** Changing the endpoint URL doesn't require rewriting the entire blob.
2. **Android SharedPreferences efficiency:** Small key-value pairs are more efficient than parsing a large JSON string on every read.

### 4.4 Encryption Considerations

**Decision: No client-side encryption for MVP. Rely on Android's keystore-backed encrypted storage.**

Rationale:
- Capacitor Preferences on Android uses SharedPreferences, which is accessible to root users but protected from other apps.
- For the auth token specifically, we should use Capacitor's `Preferences` with the `group` option to store in a more secure location (future enhancement).
- The `authToken` field is the only sensitive value. Over-engineering encryption for MVP adds complexity with marginal security benefit — if someone has root access to the phone, they have access to everything.
- **Auth tokens MUST NOT be logged or exposed in error messages** (per spec REQ-03).

### 4.5 Cold Start Flow

```
App Launch
    │
    ▼
mobile-entry.ts → detectPlatform() → platform = 'mobile'
    │
    ├─► Load SessionData from StorageAdapter
    │
    ▼
SessionData exists?
    │
    ├─ No → Show Connection Setup screen (enter URL + auth)
    │
    └─ Yes → Attempt liveness check against stored endpoint
              │
              ├─ Alive → Connect SSE, restore session
              └─ Unreachable → Show connection error, offer retry
```

## 5. Connection State Zustand Store

### 5.1 Store Design

**Critical: This store MUST follow the project's performance rules from AGENTS.md.**
- Leaf selectors, not broad subscriptions.
- Store split by change frequency — connectivity state changes frequently during reconnection.
- No unnecessary reference creation on updates.

```typescript
// packages/ui/src/stores/useConnectionStore.ts

import { create } from 'zustand';

export type ConnectionState = 'connected' | 'reconnecting' | 'disconnected';
export type NetworkType = 'wifi' | 'cellular' | 'unknown' | 'none';

interface ConnectionStateSlice {
  // High-frequency state (changes during reconnection attempts)
  status: ConnectionState;
  reconnectAttempt: number;
  lastError: string | null;
}

interface ConnectionMetaSlice {
  // Low-frequency state (changes once per connection/network change)
  lastConnectedAt: number | null;
  networkType: NetworkType;
  serverUrl: string | null;
}

interface ConnectionActionsSlice {
  setStatus: (status: ConnectionState, error?: string | null) => void;
  setNetworkType: (type: NetworkType) => void;
  setServerUrl: (url: string | null) => void;
  incrementReconnectAttempt: () => void;
  resetReconnectAttempt: () => void;
  reset: () => void;
}

type ConnectionStore = ConnectionStateSlice & ConnectionMetaSlice & ConnectionActionsSlice;

const INITIAL_STATE: ConnectionStateSlice & ConnectionMetaSlice = {
  status: 'disconnected',
  reconnectAttempt: 0,
  lastError: null,
  lastConnectedAt: null,
  networkType: 'unknown',
  serverUrl: null,
};

export const useConnectionStore = create<ConnectionStore>()((set) => ({
  ...INITIAL_STATE,

  setStatus: (status, error) =>
    set((state) => {
      // Skip no-op updates (AGENTS.md: "skip no-op updates")
      if (state.status === status && state.lastError === error) {
        return state; // Return same reference — no re-render
      }
      return {
        status,
        lastError: error ?? null,
        // Auto-set lastConnectedAt when transitioning TO connected
        ...(status === 'connected' ? { lastConnectedAt: Date.now(), reconnectAttempt: 0 } : {}),
      };
    }),

  setNetworkType: (networkType) =>
    set((state) => {
      if (state.networkType === networkType) return state;
      return { networkType };
    }),

  setServerUrl: (serverUrl) =>
    set((state) => {
      if (state.serverUrl === serverUrl) return state;
      return { serverUrl };
    }),

  incrementReconnectAttempt: () =>
    set((state) => ({ reconnectAttempt: state.reconnectAttempt + 1 })),

  resetReconnectAttempt: () => set({ reconnectAttempt: 0 }),

  reset: () => set(INITIAL_STATE),
}));
```

### 5.2 Leaf Selector Patterns

**CRITICAL: Consumers MUST use leaf selectors. Never subscribe to the entire store.**

```typescript
// ✅ CORRECT — leaf selector, only re-renders when 'status' changes
const status = useConnectionStore((s) => s.status);

// ✅ CORRECT — leaf selector for specific field
const isReconnecting = useConnectionStore((s) => s.status === 'reconnecting');

// ❌ WRONG — subscribes to entire store, re-renders on ANY change
const { status, networkType } = useConnectionStore();
```

### 5.3 Store Splitting Rationale

The `useConnectionStore` is separated from the existing `useUIStore` because:
1. **Change frequency:** Connection state changes rapidly during reconnection (up to 30 retries × state transitions). UI preferences change on user interaction.
2. **Subscriber set:** Only `ConnectionStatusBar` and connection-related UI care about connectivity. The entire app subscribes to `useUIStore` for layout/tab state. Mixing them creates unnecessary re-renders.
3. **Cross-store reads:** `ConnectivityManager` reads `useUIStore.getState().eventStreamStatus` via `.getState()` (imperative, no subscription) to correlate SSE status with mobile connectivity status.

### 5.4 Integration with Existing EventStreamStatus

The existing `useUIStore` has an `EventStreamStatus` type with values: `'idle' | 'connecting' | 'connected' | 'reconnecting' | 'paused' | 'offline' | 'error'`.

**The new `useConnectionStore` is NOT a replacement for `EventStreamStatus`.** They serve different purposes:

| Aspect | `EventStreamStatus` (UI Store) | `ConnectionStore` (New) |
|--------|--------------------------------|------------------------|
| Scope | SSE stream state | End-to-end mobile connectivity |
| Updates | Via event pipeline callbacks | Via Capacitor + liveness checks |
| Where used | SSE reconnection logic, global status | ConnectionStatusBar, mobile-specific UI |
| Desktop/Web | Always active | Always `'connected'` (no-op) |

On mobile, `ConnectivityManager` **correlates** both: if `EventStreamStatus === 'connected'` AND liveness check passes → `ConnectionStore.status = 'connected'`. If SSE drops but network is still up → `ConnectionStore.status = 'reconnecting'`.

## 6. ConnectionStatusBar Component Design

### 6.1 Component Architecture

```
┌─ ConnectionStatusBar ──────────────────────────────────────┐
│  (React.memo, leaf selectors only)                         │
│                                                             │
│  ┌─ StatusDot ─┐  ┌─ StatusText ─┐  ┌─ RetryButton ─┐    │
│  │ (memo)       │  │ (memo)        │  │ (cond. render)  │    │
│  │ green/amber/ │  │ '' / Recon... │  │ shown only when │    │
│  │ red circle   │  │ / Disconnected│  │ disconnected    │    │
│  └──────────────┘  └───────────────┘  └─────────────────┘    │
│                                                             │
│  Height: fixed 32px (no layout shift between states)        │
│  Background: theme-aware bar                                 │
└─────────────────────────────────────────────────────────────┘
```

```typescript
// packages/ui/src/components/ConnectionStatusBar.tsx

import React from 'react';
import { useConnectionStore } from '@/stores/useConnectionStore';

// Leaf selectors — components only re-render when THEIR specific data changes
const useConnectionStatus = () => useConnectionStore((s) => s.status);
const useConnectionError = () => useConnectionStore((s) => s.lastError);

export const ConnectionStatusBar = React.memo(function ConnectionStatusBar() {
  const status = useConnectionStatus();

  if (status === 'connected') {
    // Minimal: green dot only, no text (per spec SC-04a)
    return (
      <div className="connection-bar connection-bar--connected" role="status" aria-label="Connected">
        <StatusDot color="connected" />
      </div>
    );
  }

  return (
    <div className="connection-bar connection-bar--troubled" role="status" aria-live="polite">
      <StatusDot color={status === 'reconnecting' ? 'amber' : 'red'} />
      <StatusText status={status} />
      {status === 'disconnected' && <RetryButton />}
    </div>
  );
});

const StatusDot = React.memo(function StatusDot({ color }: { color: 'connected' | 'amber' | 'red' }) {
  return (
    <span
      className={`connection-dot connection-dot--${color}`}
      aria-hidden="true"
    />
  );
});

const StatusText = React.memo(function StatusText({ status }: { status: 'reconnecting' | 'disconnected' }) {
  const error = useConnectionError();
  return (
    <span className="connection-text">
      {status === 'reconnecting' ? 'Reconnecting...' : error ?? 'Disconnected'}
    </span>
  );
});

const RetryButton = React.memo(function RetryButton() {
  return (
    <button
      className="connection-retry-btn"
      onClick={() => {
        // Triggered through ConnectivityManager.reconnect()
        useConnectionStore.getState().setStatus('reconnecting');
        // connectivityManager reference injected via context or global
        getConnectivityManager()?.reconnect();
      }}
    >

    </button>
  );
});
```

## 9. S24 Ultra Specific Considerations

### 9.1 Display

- **Resolution:** 3120×1440 (19.5:9 aspect ratio, ~517 PPI)
- **CSS viewport:** After device pixel ratio (~3.5x), the CSS viewport is approximately ~891×411px
- **Safe areas:** Must respect `env(safe-area-inset-*)` for the camera cutout and rounded corners
- **Touch targets:** Minimum 44×44px per Material Design, 48×48px preferred for primary actions

### 9.2 120Hz Refresh Rate

- Android WebView does NOT cap at 60fps by default on S24 Ultra
- CSS animations will run at native refresh rate
- SSE streaming updates (via React state changes) will render at display rate
- No special CSS or JS needed to enable 120Hz — just avoid `will-change: transform` abuse on scroll containers

### 9.3 S-Pen

- S-Pen works through standard `pointer events` (`pointerdown`, `pointermove`, `pointerup`) — no custom native plugin needed
- Hover states work via `@media (pointer: fine) and (hover: hover)` — provides visual feedback when S-Pen is near screen
- Pressure sensitivity is NOT supported via web APIs, and is explicitly out of scope (per proposal)

### 9.4 Landscape

- In landscape, the S24 Ultra has a wide viewport (~891px landscape width)
- The chat + sidebar layout should show both panels
- Terminal and file tree should reflow into horizontal layout
- ConnectionStatusBar stays at the top, just shorter (28px instead of 32px)

## 10. Error Handling Strategy

### 10.1 Connectivity Error Categories

| Category | Example | User-Facing | Auto-Recovery |
|----------|---------|-------------|---------------|
| **Transient** | Network flicker, brief packet loss | "Reconnecting..." with animated dot | Yes — exponential backoff |
| **Network Change** | Wi-Fi → mobile data | "Reconnecting..." | Yes — immediate reconnect on new network |
| **Server Unreachable** | Server down, tunnel expired | "Disconnected" + Retry button | Semi: auto-retry with backoff |
| **Authentication** | Token expired, invalid credentials | "Disconnected" + settings prompt | No: requires manual re-auth |
| **Background Resume** | App was backgrounded >30s | "Reconnecting..." → liveness check | Yes: validate then restore or reconnect |

### 10.2 Error Recovery Matrix

```
┌──────────────────────┬─────────────────────────┬──────────────────────────────┐
│ Error Type           │ Immediate Action         │ Recovery Strategy            │
├──────────────────────┼─────────────────────────┼──────────────────────────────┤
│ SSE stream error     │ Set reconnecting          │ Backoff retry                │
│ Heartbeat timeout    │ Abort SSE, set reconnect  │ Backoff retry                │
│ Network lost         │ Set disconnected          │ Wait for network restore     │
│ Network change       │ Set reconnecting          │ Reset backoff, reconnect     │
│ Background resume    │ Set reconnecting          │ Liveness check → resume/fresh│
│ Liveness fail        │ Set disconnected          │ Backoff retry                │
│ Auth failure (401)   │ Set disconnected          │ Show re-auth prompt          │
│ Server error (5xx)   │ Set reconnecting          │ Backoff retry                │
└──────────────────────┴─────────────────────────┴──────────────────────────────┘
```

### 10.3 Error Propagation

- `ConnectivityManager` is the SOLE source of truth for connection state.
- `useConnectionStore` reflects the manager's state via callbacks (not polling).
- `event-pipeline.ts` reports stream events to the manager via hooks (`onStreamConnected`, `onStreamError`).
- UI components only subscribe to `useConnectionStore` leaf selectors — they never call the manager directly (except `RetryButton`, which calls `manager.reconnect()` through a reference).

### 10.4 Retry Button Behavior

When the user taps "Retry":
1. Reset exponential backoff to attempt 0.
2. Set state to `reconnecting`.
3. Attempt liveness check first (if it fails, skip to reconnect).
4. If liveness passes, restore SSE stream.
5. If liveness fails, start fresh connection from stored credentials.

## 12. Risk Mitigations in Design

| Risk (from Proposal) | Design Mitigation |
|---|---|
| SSE tunnel dies in background | Liveness check on resume validates before reconnecting; exponential backoff for persistent failures |
| Network transition drops session | `ConnectivityManager` listens to `Network` plugin, resets backoff, starts immediate reconnect on new network |
| Zombie sessions after reconnect | Liveness validation (GET /health) before restoring SSE; discard stale state if liveness fails |
| DOM performance (terminal/chat) | Leaf selectors in Zustand; React.memo on ConnectionStatusBar; CSS animations off main thread |
| Android WebView quirks | Platform detection isolates mobile code; CSS safe-area insets respect WebView behavior |
| Layout shift on state change | Fixed 32px bar height; overflow hidden; theme tokens for colors |
| Store render cascades | Separate `useConnectionStore` from `useUIStore`; leaf selectors only |
| Breaking existing web/desktop | Optional hooks in event pipeline; adapter pattern for all Capacitor APIs; platform detection gate |
